# qq -- Design Document

qq is a local markdown search engine. A single Go binary provides both a CLI
and a web frontend for full-text searching and reading markdown files stored on
disk.

## Goals

- **Fast local search** over thousands of markdown files across multiple
  directories, powered by a Bleve full-text index.
- **Dual interface**: a `qq search` CLI for quick terminal lookups and a
  `qq serve` web UI for browsing and reading.
- **Heterogeneous sources**: transparently handle markdown files with different
  frontmatter conventions (kb-style articles and Obsidian notes).
- **Incremental indexing on restart**: detect added, removed, and changed files
  and update the index accordingly, without requiring a full rebuild.

## Non-goals

- Real-time file watching (planned for the future; for now, re-index on
  restart).
- Crawling or fetching content from the web. qq only reads local files.
- Editing or writing markdown files.

## Commands

### `qq search <query>`

Runs the startup scan and index reconciliation, executes a Bleve query string
search, prints results to stdout, and exits.

Output format:

```
3 results (12ms)

  Crisis of Confidence
  instapaper | tech | 1 day ago
  ...highlighted fragment...

  A forty-year career.
  instapaper | manage | 2 months ago
  ...highlighted fragment...
```

### `qq serve`

Runs the startup scan and index reconciliation, then starts an HTTP server
(default port 8080, configurable with `--port`).

Routes:

| Path              | Description                              |
|-------------------|------------------------------------------|
| `GET /`           | Home page with search box                |
| `GET /search?q=вҖҰ` | Search results page                      |
| `GET /read/{id}`  | Render a document as HTML                |
| `GET /assets/вҖҰ`   | Static files (CSS, favicon)              |
| `GET /health`     | Health check                             |

The web UI uses server-side rendered HTML templates with Bootstrap 5 and
automatic dark mode support. The read page converts markdown to sanitized HTML
using goldmark and bluemonday.

## Configuration

qq reads a YAML config file, defaulting to `~/.config/qq/config.yaml`,
overridable with `--config`:

```yaml
directories:
  - path: ~/priv/kb/articles
  - path: ~/priv/obsidian/content
index_path: ~/.local/share/qq/index
```

- `directories` (required): list of directories to scan for `.md` files.
- `index_path` (optional): where to store the Bleve index. Defaults to
  `~/.local/share/qq/index`.

## Document Model

Each markdown file is represented as a `Document`:

```go
type Document struct {
    ID       string    // stable identifier: relative path within its directory
    Path     string    // absolute filesystem path
    Source   string    // from frontmatter or directory name
    Title    string    // from frontmatter or filename
    Author   string    // optional, from frontmatter
    URL      string    // optional, from frontmatter
    Tags     []string  // from frontmatter
    Updated  time.Time // best available timestamp
    Excerpt  string    // first N characters of body text
    Contents string    // raw markdown body (after frontmatter)
}
```

The `ID` is the relative path from the configured directory root (e.g.
`2026/w14/crisis-of-confidence.md`). This makes IDs stable across machines as
long as the directory structure is the same.

## Frontmatter Formats

qq supports two frontmatter conventions, auto-detected from the same YAML
`---` delimiters. A single `rawFrontmatter` struct with all-optional fields
handles both.

### kb-style

Generated by the markdowner tool. Always has `title`, `url`, and `saved`.
Optional fields: `author`, `source`, `date`, `tags`.

```yaml
title: A forty-year career.
author: Will Larson
url: https://lethain.com/forty-year-career/
source: instapaper
date: 2019-10-08T06:00:00-07:00
saved: 2026-01-11T07:47:12Z
tags:
    - manage
```

### Obsidian-style

No `title` field; the title is derived from the filename. Optional fields:
`tags`, `updated` (date string, `YYYY-MM-DD`).

```yaml
tags:
  - ai
updated: '2016-06-15'
```

Some Obsidian files have no frontmatter at all.

### Unified Parsing Rules

| Field     | Resolution order                                            |
|-----------|-------------------------------------------------------------|
| `Title`   | frontmatter `title` > filename without `.md`                |
| `Updated` | `saved` > `date` > `updated` > file mtime                   |
| `Author`  | frontmatter `author` (may be empty)                         |
| `Tags`    | frontmatter `tags` (may be empty)                           |
| `URL`     | frontmatter `url` (may be empty)                            |
| `Source`  | frontmatter `source` (may be empty)                         |

Files with no frontmatter get their title from the filename and their timestamp
from the file's modification time.

## Scanning and Indexing

### Scanning

On startup, the scanner walks each configured directory recursively, collects
all `.md` files, reads their content, parses frontmatter, and produces a slice
of `Document` values.

### Index Reconciliation

The indexer reconciles the scanned documents against the existing Bleve index:

1. Open or create the Bleve index at the configured path.
2. Collect all document IDs currently in the index.
3. Compute the diff:
   - **New files** (ID not in index): add to index.
   - **Removed files** (ID in index but not on disk): delete from index.
   - **Changed files** (file mtime newer than indexed timestamp): re-index.
   - **Unchanged**: skip.
4. Apply changes in a batch.

This runs synchronously before the search or server starts.

### Bleve Field Mapping

| Field      | Bleve type | Indexed | Stored |
|------------|------------|---------|--------|
| `title`    | text       | yes     | yes    |
| `contents` | text       | yes     | yes    |
| `tags`     | keyword    | yes     | yes    |
| `url`      | keyword    | no      | yes    |
| `source`   | keyword    | yes     | yes    |
| `author`   | text       | yes     | yes    |
| `updated`  | datetime   | yes     | yes    |
| `excerpt`  | text       | no      | yes    |

## Architecture

```
qq/
  cmd/qq/
    main.go              CLI entry point, cobra subcommands
  internal/
    config/
      config.go          Load and validate YAML config
    document/
      document.go        Document type, frontmatter parsing
    scanner/
      scanner.go         Walk directories, produce []Document
    index/
      index.go           Bleve wrapper: create, reconcile, search
    markdown/
      markdown.go        Markdown to HTML, markdown to plain text
    serve/
      server.go          HTTP server, routing
      api.go             Search/read logic backed by the index
      dynamic.go         Template rendering (home, search, read)
      static.go          Static file serving
      template/
        template.go      Template engine with optional hot reload
      ui/
        templates/       HTML templates (home, search, read, partials)
        assets/          CSS, favicon
```

## Dependencies

- `github.com/blevesearch/bleve/v2` -- full-text search index
- `github.com/yuin/goldmark` -- markdown to HTML conversion
- `github.com/microcosm-cc/bluemonday` -- HTML sanitization
- `gopkg.in/yaml.v3` -- YAML frontmatter and config parsing
- `github.com/spf13/cobra` -- CLI subcommand framework
- Standard library `log/slog` -- structured logging
- Standard library `net/http` and `html/template` -- web server (Go 1.22+
  routing, no third-party router needed)

## Future Work

- **File watching**: use fsnotify to re-index on file changes without restart.
- **Additional CLI commands**: `qq list`, `qq read <path>`.
- **Tag browsing**: dedicated tag index page in the web UI.
- **Pagination**: paginate search results beyond the initial 20.
